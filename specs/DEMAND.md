# Routing Demand Generation

This document describes the precise system by which SimTower generates
routing demand each day — when entities decide to travel, where they go,
and how the staggering creates realistic traffic patterns.

## Overview

Routing demand is not generated by a central scheduler. Each sim
has a **gate handler** (a per-tick refresh function) that probabilistically
decides whether to call the **dispatch handler** (the routing function).
The gate handler reads `g_daypart_index`, `g_day_tick`, RNG output, and
entity-specific state to make this decision.

Because entities are serviced in 1/16th stride order (one tick services
entity indices `tick % 16, tick + 16, tick + 32, ...`), each entity gets
one gate check every 16 ticks. The combination of daypart windows and
per-tick probability creates staggered demand waves, not instantaneous
spikes.

## Key Globals

| Address | Name | Type | Meaning |
|---------|------|------|---------|
| `1288:bb8b` | `g_daypart_index` | byte | `g_day_tick / 400`, range 0–6 |
| `1288:bb8a` | `g_calendar_phase_flag` | byte | `((day_counter % 12) % 3) >= 2` |
| `1288:bc52` | `g_day_tick` | word | 0–2599 |

## Stochastic Gating

Gate handlers use the global RNG (`CALLF 0x1000:324d`) to implement
per-tick probability checks. The pattern is:

```
rand() % N == 0   →  dispatch (1/N chance per entity tick)
```

**Critical correction**: the current spec describes several gates as
`day_counter % N == 0` (deterministic, fires on specific days). The binary
uses `rand() % N == 0` (stochastic, fires independently each entity tick
with probability 1/N). These are fundamentally different:

- `rand() % 12 == 0`: ~1/12 chance per entity service tick (~1/192 chance per game tick for a given entity, since stride is 16)
- `rand() % 6 == 0`: ~1/6 chance per entity service tick

The `day_counter` variable is never read in any gate handler.

## Family 7 — Office Workers

6 workers per office, `occupant_index` 0–5. The six runtime entities are created when
the office is placed, before the office is rented. They start in state `0x20`; rental and
elevator demand are produced only when one of those existing workers passes the gate and
the route resolver accepts a lobby-to-office route.

### Daily Lifecycle

```
  0x20 ──(service request)──> 0x60 ──> 0x21 ──> ...
    │                                    │
  0x00 ──(lobby→office)──> 0x40 ──> 0x21 ──(arrived)──> advance
    │                                                      │
    │                                              0x01 ──(office→venue)──> 0x41 ──> 0x42 ──> 0x22/0x23
    │                                                                                             │
  0x05 ──(office→lobby)──> 0x45 ──────────────────────────────────────────────────────────────> done
    ↑                                                                                              │
    └──────────── daypart >= 4 forces 0x05 from states 0x00, 0x01, 0x02 ──────────────────────────┘
```

Night: states 0x25/0x26/0x27 park until `g_day_tick > 2300`, then → 0x20. The daily
runtime reset parks office workers in `0x27` and clears route fields; it does not respawn
them at the office floor or lobby floor.

`0x4x` and `0x6x` are in-transit state aliases, not a generic "elevator bit." Runtime byte
`+8` determines whether the in-transit state is continuing on a direct route (`+8 < 0x40`)
or in the queued-car path (`+8 >= 0x40`).

### Gate Table (Binary-Verified)

Gate address: `1228:1cb5` (refresh handler), jump table at `1228:2005`.

| State | Gate condition | Action |
|-------|---------------|--------|
| **0x00** | daypart >= 4 | force state → 0x05 |
| | occupant 0, daypart 0 | 1/12 chance → dispatch |
| | occupant 0, dayparts 1–3 | always dispatch |
| | occupant != 0, dayparts 0–2 | no dispatch |
| | occupant != 0, daypart 3 | 1/12 chance → dispatch |
| **0x01, 0x02** | daypart >= 4 | force state → 0x05 |
| | daypart 0 | no dispatch |
| | daypart 1 | 1/12 chance → dispatch |
| | dayparts 2–3 | always dispatch |
| **0x05** | daypart 4 | 1/6 chance → dispatch |
| | dayparts 5–6 | always dispatch |
| | daypart < 4 | no dispatch |
| **0x20** | `calendar_phase_flag != 0` | no dispatch (blocked) |
| | `occupied_flag == 0` | no dispatch (required) |
| | daypart 0 | 1/12 chance → dispatch |
| | dayparts 1–2 | always dispatch |
| | daypart >= 3 | no dispatch; state remains `0x20` |
| **0x21** | daypart >= 4 | force state → 0x27 + release service request |
| | daypart 3 | 1/12 chance → dispatch |
| | dayparts 0–2 | no dispatch |
| **0x22, 0x23** | daypart >= 4 | force state → 0x27 + release service request |
| | dayparts 2–3 | always dispatch |
| | dayparts 0–1 | no dispatch |
| **0x25, 0x26, 0x27** | `g_day_tick > 2300` | force state → 0x20 |
| | otherwise | parked (no dispatch) |

### Dispatch Table

Dispatch address: `1228:2031`, jump table at `1228:2aac`. States `0x4x`
share handlers with `0x0x`; states `0x6x` share with `0x2x`.

| State | Route | Direction | Notes |
|-------|-------|-----------|-------|
| 0x00/0x40 | lobby (floor 0, EXE raw floor 10) → office floor | inbound | morning commute |
| 0x01/0x41 | office → commercial venue | outbound | selector 2 (fast food) |
| 0x02/0x42 | continue venue transit | — | resolve to saved zone index |
| 0x05/0x45 | office floor → lobby (floor 0, EXE raw floor 10) | outbound | evening commute |
| 0x20/0x60 | assign service request, route lobby → office floor | inbound | eval-gated rental/opening path |
| 0x21/0x61 | route to lobby or saved floor | outbound | late-day gate parks directly to `0x27`, not through `0x05` |
| 0x22/0x62 | release venue, route home | outbound | |
| 0x23/0x63 | 16-tick venue dwell, then route to saved target | — | |

Route-result state writes for the core office commute states:

| State family | Resolver result | Next state / side effect |
|--------------|-----------------|--------------------------|
| `0x20/0x60` | `-1` and office still vacant (`unit_status >= 0x10`) | release service request, write `0x20`, clear route timing fields |
| `0x20/0x60` | `-1` and office already open (`unit_status < 0x10`) | write `0x25` parked/intermediate state |
| `0x20/0x60` | `0`, `1`, or `2` | if vacant, call office activation and write `unit_status = 0`; write worker state `0x60` |
| `0x20/0x60` | `3` | if vacant, call office activation and run the immediate-arrival branch |
| `0x00/0x40` | `-1` | write `0x26` and release service request |
| `0x00/0x40` | `0`, `1`, or `2` | write `0x40` |
| `0x00/0x40` | `3` | write `0x21` |
| `0x21/0x61` | `-1` | write `0x26` and release service request |
| `0x21/0x61` | `0`, `1`, or `2` | write `0x61` |
| `0x21/0x61` | `3` | call `advance_office_presence_counter`, write `0x05` |
| `0x05/0x45` | `-1` | write `0x26` and release service request |
| `0x05/0x45` | `0`, `1`, or `2` | write `0x45` |
| `0x05/0x45` | `3` | write `0x27` and release service request |
| `0x00/0x40` | *(all results, first dispatch only)* | call `decrement_office_presence_counter` — fires when base state is `0x00`, not `0x40` |
| `0x01/0x41` | *(all results, first dispatch only)* | call `decrement_office_presence_counter` — fires when base state is `0x01`, not `0x41` |
| `0x02/0x42` | *(all results, first dispatch only)* | call `decrement_office_presence_counter` — fires when base state is `0x02`, not `0x42` |
| `0x05/0x45` | *(all results, first dispatch only)* | call `decrement_office_presence_counter` — fires when base state is `0x05`, not `0x45` |

### Demand Profile

**Morning wave** (dayparts 0–3):
- Occupant 0 starts trickling in from daypart 0 (1/12 per tick), flowing steadily in dayparts 1–3
- Occupants 1–5 only dispatch in daypart 3 (1/12 per tick), creating a concentrated midday arrival
- Effect: one early commuter per office, followed by a burst of 5 workers around midday

**Lunch wave** (dayparts 1–3):
- Commercial trips start in daypart 1 (1/12) and intensify through dayparts 2–3
- Each worker makes one trip to a fast-food venue and returns

**Evening wave** (dayparts 4–6):
- Any worker still in morning states (0x00/0x01/0x02) is forced to state 0x05
- 0x05 dispatches with 1/6 in daypart 4, always in dayparts 5–6
- Effect: gradual evening departure with most leaving by daypart 5
- After same-floor evening arrival at the lobby, the worker is parked in state `0x27`.
  The binary does not need a separate "worker is currently in office" boolean for the next
  morning commute; the next morning route originates from the lobby floor when the state
  machine reaches a lobby-to-office state.

## Families 3/4/5 — Hotel Room Occupants

Family 3 = single (1 occupant), family 4 = twin (2), family 5 = suite (3).

```
  0x20 ──(lobby→room)──> 0x60 ──> activate ──> 0x01
    │                                            │
    │                                   0x01 ──(room→venue)──> 0x41

    │                               (arrived)                 (arrived)
    │                                     │                      │
```

### Dispatch Table

Dispatch address: `1228:2dae`, jump table at `1228:3520`.

| State | Route | Direction | Notes |
|-------|-------|-----------|-------|
| 0x20/0x60 | lobby → room floor | inbound | assigns room on first entry |
| 0x01/0x41 | room → commercial venue | outbound | decrements unit_status |
| 0x22/0x62 | release venue, route back to room | inbound | |
| 0x04 | sibling sync check | — | no route, state bookkeeping only |
| 0x10 | checkout counter dispatch | — | rewrites unit_status countdown |
| 0x05/0x45 | room → lobby (floor 0) | outbound | checkout trip |

### Demand Profile

**Check-in wave** (daypart 4+):
- Guests route from lobby to assigned rooms
- Gated by `pairing_pending_flag` (room was claimed by housekeeping entity)
- Daypart 4: trickle (1/12), daypart 5 tick < 2300: steady flow

**Venue trip** (daypart 4 only):
- After arriving at room, guests may visit a commercial venue
- 1/6 chance per tick in daypart 4 only
- If daypart 5 arrives before dispatch, forced to sibling sync

**Checkout** (dayparts 0–5):
- State 0x10 dispatches freely in dayparts 0–4
- State 0x05 (actual checkout route) dispatches dayparts 1–5, 1/12 in daypart 0
- Suppressed in daypart 6

| State | Route | Direction | Notes |
|-------|-------|-----------|-------|

| 0x01/0x41 | → commercial venue | outbound | selector from `resident_index % 4` |
| 0x20/0x60 | unsold condo sale outbound | outbound | sale fires on successful route |

### Demand Profile

**Morning cycle** (dayparts 0–4):
- State 0x10 dispatches freely in dayparts 0–4 (morning)
- Residents make outbound trips (0x00) and venue visits (0x01)
- Standard path: daypart 0 (1/12), dayparts 1–5 (always)

**Calendar-phase stagger**:
- Every 3rd day in the 12-day cycle, one resident per condo (subtype%4==0)
  delays their venue trip: can only dispatch in daypart 4 (1/6) or is forced
  to sync in daypart 5+

**Sibling sync** (daypart 5+):
- resident_index 2 syncs immediately when daypart >= 5
- Others: daypart 5 with 1/12 per tick, daypart 6 (tick > 2400) always

## Family 0x0f — Housekeeping

Transient search entity. One per hotel room slot. Gate function:
`gate_housekeeping_room_claim_state` at `1228:????`.

No daypart gating in the gate handler — the entity is serviced on every

## Families 0x24–0x28 — Cathedral Guests

5 floor slices × 8 entities = 40 cathedral guests. Spawned at checkpoint
0x000 into state 0x20.

### Gate Table (Binary-Verified)

Gate address: `1228:5b5a`. Inline if-chain.

| State | Gate condition | Action |
|-------|---------------|--------|
| **0x20** | `calendar_phase_flag != 1` | no dispatch |
| | daypart 0, tick > 80 | 1/12 chance (`rand() % 12 == 0`) → dispatch |
| | daypart 0, tick > 240 | always dispatch (guaranteed) |
| | daypart >= 1 | force state → 0x27 |
| **0x05** | always | dispatch (no gate) |

Note: the tick > 80 random check AND the tick > 240 guaranteed check both
run on the same gate invocation. An entity at tick 241+ will dispatch from
the guaranteed path even if the random check at tick 81–240 failed on
previous ticks.

### Dispatch Table

Jump table at `1228:5f29`, 4 entries.

| State | Route | Notes |
|-------|-------|-------|
| 0x20/0x60 | lobby (floor 0) → cathedral (floor 100) | outbound |
| 0x05/0x45 | cathedral (floor 100) → lobby (floor 0) | midday return (set by checkpoint 0x04b0) |

### Demand Profile

Cathedral guests only dispatch on calendar-phase days (`calendar_phase_flag == 1`),
during daypart 0 only. They trickle out from tick 81 (1/12 per tick) and
are all guaranteed dispatched by tick 241. If any haven't dispatched by
daypart 1, they are parked (0x27) and miss the window.

## Corrections to Previous Spec

The following errors in `specs/PEOPLE.md` were found by comparing the spec
against the binary disassembly:

### 1. Stochastic gating uses RNG, not day_counter

**All** probability gates in the family state machines use `rand() % N == 0`,
not `day_counter % N == 0`. The `day_counter` variable is never read in
any gate handler.

Affected entries: every mention of `day_counter % 6 == 0` or
`day_counter % 12 == 0` in the gate tables of PEOPLE.md should be
`rand() % 6 == 0` (1/6 probability) or `rand() % 12 == 0` (1/12
probability).

### 2. Office state 0x00 is staggered by occupant_index

The spec says state 0x00 dispatches uniformly in dayparts 1–3. The binary
shows `occupant_index == 0` dispatches in dayparts 0–3, while
`occupant_index != 0` only dispatches in daypart 3 (with 1/12 probability).
This creates a realistic staggered morning commute.

### 3. Office state 0x20 does NOT dispatch at daypart >= 3

The spec says "daypart ≥ 3 → dispatch". The binary shows daypart >= 3
returns without dispatch. State 0x20 only dispatches in dayparts 0–2.

### 4. Office state 0x21 does NOT dispatch at daypart >= 4

The spec says "daypart ≥ 4 → dispatch". The binary shows daypart >= 4
forces state → 0x27 and releases the service request, without calling
the dispatch handler.

### 5. Condo state 0x21 is staggered by resident_index

The spec does not document this. `resident_index == 2` dispatches from
daypart 3 (1/12) + daypart > 3 (always). Others dispatch from daypart 4
(1/12) + daypart > 4 (always).

### 6. Condo state 0x04 is staggered by resident_index

The spec does not distinguish. `resident_index == 2` always dispatches
when daypart >= 5. Others use the probability gate (1/12 when tick <= 2400).
